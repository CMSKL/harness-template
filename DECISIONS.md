# DECISIONS.md

> 记录重要设计决策。不要记录琐碎改动，只记录未来会影响实现方向的选择。

## 模板

### YYYY-MM-DD: 决策标题
- 状态: proposed / accepted / deprecated
- 背景: 为什么需要这个决策
- 决策: 做了什么选择
- 原因: 为什么选这个，不选别的
- 影响: 会影响哪些模块或后续工作
- 否决方案:
  - 方案 A：为什么没选
  - 方案 B：为什么没选

---

## 示例

### 2026-04-02: 使用功能清单驱动 agent 执行范围
- 状态: accepted
- 背景: agent 在宽泛任务下容易同时启动多个功能，导致做了一堆半成品。
- 决策: 用 `features.json` 作为唯一任务范围来源，并强制 WIP=1。
- 原因: 这样可以把“写了代码”变成“通过验证才完成”。
- 影响: 所有功能开发都必须先在功能清单里定义，再开始实现。
- 否决方案:
  - 自由形式进度记录：状态歧义太大，跨会话恢复成本高。
  - 只靠对话说明范围：无法持久化，也不能被脚本消费。
