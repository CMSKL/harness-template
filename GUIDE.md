# AgentHarness 使用指南

> 让 AI Agent 能够稳定、可靠地帮你开发项目的工程框架

---

## 1. 什么是 Harness？

### 一句话解释

**Harness（马具）是给 AI Agent 的一套"工作规范+工具包"**，就像给马配上马鞍和马缰，让它能按照你的意图稳定工作，而不是乱跑。

### 一个具体场景

想象一下这个场景：

> 你对 Claude Code 说："帮我给这个项目加一个用户登录功能。"
>
> **没有 Harness 时**：
> - Claude 花 20 分钟改了一堆文件
> - 告诉你"做完了"
> - 你一试：数据库连不上、密码没加密、测试没通过
> - 下次新开会话，Claude 忘了上次做到哪，重新开始
>
> **有了 Harness 后**：
> - Claude 先看 `CLAUDE.md` 了解项目怎么运行
> - 看 `features.json` 选一个有验证命令的功能
> - 实现功能 → 跑验证 → 通过才标记完成
> - 更新 `PROGRESS.md` 记录进度
> - 下次新开会话，读 `PROGRESS.md` 立刻知道该做什么

### Harness 的核心思想

```
┌─────────────────────────────────────────────────────────────┐
│                     Harness =                               │
├─────────────────────────────────────────────────────────────┤
│  规则文档 (CLAUDE.md)      → 告诉 Agent 什么能做、什么不能做  │
│  功能清单 (features.json)  → 告诉 Agent 要做什么、怎么算做完  │
│  进度跟踪 (PROGRESS.md)    → 告诉 Agent 现在做到哪了         │
│  验证命令 (Makefile)       → 告诉 Agent 怎么检查做得对不对    │
│  质量记录 (QUALITY.md)     → 告诉 Agent 哪里需要优先改进      │
└─────────────────────────────────────────────────────────────┘
```

**简单说：把你要交代的事情写下来，放在 Agent 能看到的地方。**

## 2. 这套模板解决什么问题？

### 问题 1：Agent 不知道项目是什么

**场景**：你把一个项目丢给 Agent，它问"这是什么？"你解释了 10 分钟。

**解决**：`CLAUDE.md` 写好项目描述，Agent 自己看。

### 问题 2：Agent 一次做太多，最后都没做完

**场景**：你说"加登录功能"，Agent 同时改数据库、写 API、做前端，2 小时后每个都是半成品。

**解决**：`WIP=1` 规则 + `features.json`，一次只做一件事，做完验证通过才算完。

### 问题 3：Agent 说"做完了"，但实际跑不通

**场景**：Agent 说功能做完了，你一测试，根本不能用。

**解决**：每个功能必须有 `verification` 命令，跑通才能标记 passing。

### 问题 4：新会话接不住上次的工作

**场景**：昨天做到一半，今天新开会话，Agent 完全不知道昨天做了什么。

**解决**：`PROGRESS.md` 记录进度，新会话先读它。

### 问题 5：项目规则全靠口头交代

**场景**：每次都要提醒"别忘了写测试""记得跑 lint"。

**解决**：规则写在 `CLAUDE.md` 和 `docs/` 里，Agent 每次都会看。

### 问题 6：不知道哪里做得不好

**场景**：项目越来越乱，但不知道具体哪里有问题。

**解决**：`QUALITY.md` 给每个模块打分，D 级的优先治理。

## 3. 核心概念图解

### 3.1 一个功能从想法到完成的流程

```
你的想法
    │
    ▼
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  功能清单     │ →  │   实现功能    │ →  │   验证通过    │
│ features.json │    │              │    │              │
│              │    │ 写代码        │    │ 跑验证命令    │
│ 描述: 用户    │    │ 写测试        │    │ 通过=passing │
│ 可以注册      │    │              │    │ 不通过=继续改 │
│ 验证: curl    │    │              │    │              │
│ 命令          │    │              │    │              │
└──────────────┘    └──────────────┘    └──────────────┘
       │                                      │
       ▼                                      ▼
  state: not_started                    state: passing
       │                                      │
       └──────── 做完一件再做一件 ────────────┘
```

### 3.2 会话开始和结束的固定动作

**每次会话开始，Agent 应该：**

```
1. 读 CLAUDE.md    → 了解项目是什么
2. 读 PROGRESS.md  → 了解现在做到哪
3. 读 features.json → 了解还有哪些功能要做
4. 跑 make check   → 确认项目状态正常
5. 从 PROGRESS.md 的"下一步"开始工作
```

**每次会话结束，Agent 必须：**

```
1. 更新 PROGRESS.md    → 记录今天做了什么
2. 更新 features.json  → 如果有功能状态变化
3. 跑 make check       → 确认一切正常
4. 跑 check-clean-state.sh → 确认没有遗留调试代码
5. git commit          → 提交所有变更
```

### 3.3 三层验证（完成的标准）

一个功能要标记"完成"，必须通过三层验证：

```
┌────────────────────────────────────────────────────────────┐
│ Level 1: 静态检查                                          │
│  make lint && make typecheck                               │
│  代码格式对不对、类型有没有错                               │
└────────────────────────────┬───────────────────────────────┘
                             │ 不通过就停
                             ▼
┌────────────────────────────────────────────────────────────┐
│ Level 2: 单元/集成测试                                     │
│  make test                                                 │
│  功能逻辑对不对、模块协作是否正常                           │
└────────────────────────────┬───────────────────────────────┘
                             │ 不通过就停
                             ▼
┌────────────────────────────────────────────────────────────┐
│ Level 3: 端到端验证                                        │
│  功能特定的验证命令（如 curl 测试 API）                     │
│  用户实际使用场景能不能跑通                                 │
└────────────────────────────────────────────────────────────┘
                             │
                             ▼
                     标记为 passing（真正完成）
```

**规则：L1 没过不能进 L2，L2 没过不能进 L3。**

### 3.4 功能清单状态机

```
┌─────────────┐     选择开始做      ┌───────────┐
│ not_started │ ─────────────────→ │  active   │
│   未开始     │                    │  正在做    │
└─────────────┘                    └─────┬─────┘
       ▲                                 │
       │         验证通过                │ 发现阻塞
       └───────────────────────────────  │
                                          ▼
                                   ┌───────────┐
                                   │  blocked  │
                                   │  被阻塞    │
                                   └───────────┘
                                          │
                                          │ 解决阻塞
                                          ▼
                                   ┌───────────┐
                                   │  passing  │
                                   │  已完成    │
                                   └───────────┘
```

**关键规则**：
- 任何时候只能有一个 `active`（WIP=1）
- `passing` 是单向的，验证通过就不能退回
- 只有 `verify-feature.sh` 能把 `active` 改成 `passing`

## 4. 文件结构一览

```
agentharness/
│
├── 📋 CLAUDE.md              ← Agent 的"入职手册"
│   项目是什么、怎么启动、哪些规则不能违反
│
├── 📊 PROGRESS.md            ← "工作日志"
│   今天做了什么、明天做什么、被什么卡住
│
├── 📝 DECISIONS.md           ← "决策记录"
│   为什么用这个技术、为什么这样设计
│
├── ⭐ features.json          ← "任务清单"
│   有哪些功能、怎么算做完、现在做到哪
│
├── 🎯 QUALITY.md             ← "质量评分表"
│   哪个模块健康、哪个模块需要治理
│
├── 🔧 Makefile               ← "快捷命令"
│   make setup / make test / make check
│
├── 📖 GUIDE.md               ← "你在看的这份文档"
│
├── docs/                     ← "详细规则手册"
│   ├── architecture.md       架构设计
│   ├── api-patterns.md       API 规范
│   ├── database-rules.md     数据库规则
│   ├── testing-standards.md  测试标准
│   ├── constraints.md        硬约束清单
│   ├── sprint-contract-template.md  任务开始前写
│   └── handoff-template.md   会话交接用
│
├── scripts/                  ← "自动化脚本"
│   ├── session-start.sh      会话开始时跑
│   ├── session-end.sh        会话结束时跑
│   └── check-clean-state.sh  检查是否干净
│
├── src/                      ← 你的源代码（初始化后创建）
└── tests/                    ← 你的测试文件（初始化后创建）
```

## 5. 快速上手：一个完整的例子

### 场景

假设你要做一个"待办事项"应用，想让 AI Agent 帮你开发。

### 第一步：初始化项目（第 1 个会话）

**1. 复制模板**
```bash
cp -r agentharness/ my-todo-app/
cd my-todo-app
```

**2. 填写 CLAUDE.md**

```markdown
## 项目概览
- **项目名称**：MyTodo
- **项目类型**：Web 后端 API
- **一句话描述**：一个支持用户注册、登录、创建待办事项的 REST API
- **技术栈**：Node.js 20 + Express + SQLite
- **代码规模**：约 2000 行，5 个模块

## 快速开始
make setup    # 安装依赖
make dev      # 启动开发服务器
make test     # 运行测试
make check    # 完整验证

## 硬约束
1. 所有 API 必须返回 JSON 格式
2. 所有数据库操作必须通过 ORM，禁止裸写 SQL
3. 所有路由必须写测试
```

**3. 配置 Makefile**

```makefile
setup:
	npm install

dev:
	npm run dev

test:
	npm test

typecheck:
	npx tsc --noEmit

lint:
	npx eslint src/

build:
	echo "No build step for Node.js"

check: lint typecheck test
	@echo "All checks passed"
```

**4. 填写 features.json**

```json
{
  "schema_version": "1.0",
  "project": "MyTodo",
  "rules": {
    "wip_limit": 1,
    "single_source_of_truth": true,
    "passing_requires_verification": true
  },
  "features": [
    {
      "id": "F01",
      "behavior": "用户可以注册账户",
      "verification": "curl -X POST http://localhost:3000/api/register -d '{\"email\":\"test@example.com\",\"password\":\"123456\"}' | grep 201",
      "state": "not_started",
      "priority": 1,
      "dependencies": []
    },
    {
      "id": "F02",
      "behavior": "用户可以登录",
      "verification": "curl -X POST http://localhost:3000/api/login -d '{\"email\":\"test@example.com\",\"password\":\"123456\"}' | grep token",
      "state": "not_started",
      "priority": 2,
      "dependencies": ["F01"]
    },
    {
      "id": "F03",
      "behavior": "用户可以创建待办事项",
      "verification": "curl -X POST http://localhost:3000/api/todos -H 'Authorization: Bearer TOKEN' -d '{\"title\":\"测试任务\"}' | grep 201",
      "state": "not_started",
      "priority": 3,
      "dependencies": ["F02"]
    }
  ]
}
```

**5. 填写 PROGRESS.md**

```markdown
## 当前状态
- 最新 commit: (init)
- 当前分支: main
- 构建状态: 未初始化
- 测试状态: 未初始化
- 当前激活功能项: none

## 已完成
- [x] 项目初始化

## 进行中
- [ ] 环境配置（Makefile 已配置）

## 下一步
1. 安装依赖并跑通首个测试
2. 实现 F01：用户注册
```

**6. 提交**

```bash
git add .
git commit -m "init: 项目初始化"
```

### 第二步：开发第一个功能（第 2 个会话）

**对 Agent 说：**

> 请帮我实现 F01 "用户可以注册账户"功能。
>
> 步骤：
> 1. 先读 CLAUDE.md 了解项目
> 2. 读 PROGRESS.md 了解当前状态
> 3. 把 features.json 中 F01 的 state 改为 active
> 4. 实现注册功能（包括测试）
> 5. 跑验证命令确认通过
> 6. 把 F01 的 state 改为 passing
> 7. 更新 PROGRESS.md
> 8. 提交代码

**Agent 的工作流程：**

```
1. 读 CLAUDE.md → 了解用 Node.js + Express + SQLite
2. 读 features.json → F01 需要 POST /api/register
3. 改 state: not_started → active
4. 写代码：
   - src/routes/auth.js（注册路由）
   - src/models/user.js（用户模型）
   - tests/auth.test.js（测试）
5. 跑验证：
   make lint         ✓
   make typecheck    ✓
   make test         ✓
   curl 验证命令    ✓
6. 改 state: active → passing
7. 更新 PROGRESS.md
8. git commit -m "feat: F01 用户注册功能"
```

### 第三步：下一个功能（第 3 个会话）

新开个会话，对 Agent 说：

> 继续开发 F02。

Agent 会：
1. 读 PROGRESS.md → 看到 F01 已完成，F02 未开始
2. 读 features.json → F02 依赖 F01（已满足）
3. 开始实现 F02...

## 6. 详细完善步骤

### 第一步：补全 CLAUDE.md（最重要）

这是 Agent 看到的第一份文件，必须让它能回答：

1. **这是什么？** → 项目类型、技术栈、规模
2. **怎么启动？** → `make setup` 和 `make dev`
3. **怎么验证？** → 三层验证层级
4. **什么不能做？** → 硬约束（不超过 15 条）

**模板：**

```markdown
## 项目概览
- **项目名称**：
- **项目类型**：Web 后端 / 前端 / CLI / 微服务
- **一句话描述**：
- **技术栈**：语言 + 框架 + 数据库
- **代码规模**：约 X 行，X 个模块

## 快速开始
make setup    # 安装依赖
make dev      # 启动开发
make test     # 运行测试
check         # 完整验证

## 硬约束（不能违反）
1. ...
2. ...
3. ...

## 功能开发规则
- WIP=1：一次只做一件事
- 完成定义：L1+L2+L3 都通过
- 不许顺手重构
```

### 第二步：配置 Makefile

把 echo 换成真实命令：

| 命令 | Node.js 示例 | Python 示例 |
|------|-------------|-------------|
| setup | `npm install` | `pip install -r requirements.txt` |
| test | `npm test` | `pytest` |
| lint | `npx eslint` | `ruff check .` |
| typecheck | `npx tsc --noEmit` | `mypy src/` |
| build | `npm run build` | `echo "no build"` |
| dev | `npm run dev` | `python main.py` |

### 第三步：填写功能清单

**功能粒度原则：**

- ✅ 好："用户可以注册"
- ❌ 太大："实现用户系统"
- ❌ 太小："创建 User 模型的 name 字段"

**每个功能必须有：**
- `behavior`：用户视角的描述
- `verification`：能执行的验证命令
- `dependencies`：依赖的其他功能 ID

### 第四步：写 PROGRESS.md

至少填这几个：

```markdown
## 当前状态
- 最新 commit: (init)
- 当前激活功能项: none

## 已完成
- [x] 项目初始化

## 进行中
- [ ] F01：用户注册

## 下一步
1. 跑通第一个测试
2. 实现 F01
```

### 第五步：写硬约束

只写真的不能违反的规则，例如：

```markdown
## 安全约束
- 禁止在代码里写死密码/密钥
- 所有 API 必须验证身份（除了登录/注册）

## 架构约束
- UI 层不能直接访问数据库
- 所有数据库操作必须通过 ORM

## 流程约束
- 所有代码必须通过 lint 才能提交
- 所有功能必须有测试
```

### 第六步：添加架构边界检查（可选）

如果你发现 Agent 经常犯某类错误，可以写个脚本自动检查。

**示例**：检查是否直接用了 `fs` 模块：

```bash
#!/bin/bash
if grep -r "require('fs')" src/ui/; then
  echo "ERROR: UI 层不能直接访问文件系统"
  exit 1
fi
```

> 💡 不会写脚本？直接问 AI："帮我写一个检查 XX 的脚本"

## 8. 常见问题和解决

### Q1: Agent 不看这些文件怎么办？

**解决**：在提示里明确要求。

不要说：
> "帮我加一个登录功能"

要说：
> "请先读 CLAUDE.md 和 PROGRESS.md，然后实现 F02 登录功能。记得跑验证命令。"

### Q2: Agent 同时做多个功能怎么办？

**解决**：强调 WIP=1。

在 CLAUDE.md 里写清楚：
> ## 功能开发规则
> - **WIP=1**：任何时刻只能有一个 active 功能
> - 当前功能验证通过才能开始下一个

### Q3: Agent 说做完但验证失败？

**解决**：强调三层验证。

在 CLAUDE.md 里写：
> ## 完成定义
> - 完成 = L1+L2+L3 全部通过
> - L1 没过不能做 L2，L2 没过不能做 L3

### Q4: 新会话不知道上次做什么？

**解决**：每次会话结束强制更新 PROGRESS.md。

在 CLAUDE.md "会话结束"部分写清楚：
> 结束前必须：更新 PROGRESS.md，记录今天做了什么、明天做什么。

### Q5: 不知道怎么写验证命令？

**解决**：验证命令可以是任何能执行的命令：

```bash
# API 测试
curl -X POST http://localhost:3000/api/login | grep 200

# 文件存在测试
test -f src/components/Login.tsx && echo "OK"

# 测试套件
npm test -- --grep "login"

# 自定义脚本
bash scripts/test-login.sh
```

## 9. 怎么判断 Harness 够不够好？

### 测试 1：冷启动测试

开个全新会话，不给任何口头说明，只让它看仓库。然后问：

1. 这是什么项目？
2. 怎么启动？
3. 怎么验证代码对不对？
4. 现在有哪些功能要做？
5. 下一步该做什么？

**5 个都能答对** = Harness 合格

### 测试 2：功能完成率

让 Agent 做 5 个功能：

- 80%+ 一次通过 = 优秀
- 60-80% = 合格，需微调
- <60% = 需要改进 Harness

### 测试 3：会话重建时间

新会话从启动到开始有效工作的时间：

- <3 分钟 = 优秀
- 3-10 分钟 = 合格
- >10 分钟 = PROGRESS.md 需要改进

## 10. 给新手的建议

### 不要一开始就填得很满

❌ 错误做法：
- 第一天就把所有文档写满
- 写很多约束规则
- 定义几十个功能

✅ 正确做法：
1. **先搭骨架**：填好 CLAUDE.md 前 50 行
2. **跑起来**：确保 `make setup` 和 `make test` 能用
3. **做第一个功能**：让 Agent 完整走一遍流程
4. **哪里出问题补哪里**：Agent 踩坑了，就在对应位置加规则
5. **定期清理**：删掉过时、重复的规则

### 渐进式完善

| 阶段 | 做什么 | 目标 |
|------|--------|------|
| 第 1 天 | 填 CLAUDE.md + Makefile + 第一个功能 | 能跑通第一个功能 |
| 第 1 周 | 添加 5-10 个功能，补关键约束 | 完成核心功能 |
| 第 1 月 | 完善专题文档，添加质量评估 | 能稳定开发 |
| 持续 | 根据问题补规则，定期清理 | 越来越好用 |

### 记住核心原则

1. **写下来**：所有规则必须写在文件里，不要口头交代
2. **能验证**：每个功能必须有明确的验证标准
3. **可追溯**：每次会话结束更新进度
4. **单向完成**：验证通过才能标记完成，不能回退
5. **一次一件**：WIP=1，做完一件再做下一件

**Harness 不是一次性的文档，而是和代码一起演化的工程系统。**
